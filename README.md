# Steam 游戏数据分析：Tag 机会发现与分析辅助系统

**🔗 在线体验地址**: [Steam 决策辅助系统 (V2)](https://steam-data-analysis-v2-lv8eu5t3vi98rr2mjnk2ck.streamlit.app/)

本项目通过深度挖掘 Steam 全量游戏数据，帮助游戏团队在**赛道研究、竞品分析等环节**，科学地验证“品类方向”与“Tag 组合”的市场潜力。

---

## 1. 为什么要做这个项目？ 
目前市面上关于 Steam 游戏的数据平台（如 SteamDB、SteamSpy 等）以及开源数据集（如 Kaggle 上的一些）虽然信息丰富，但在实际的业务场景中，仍存在明显的痛点：
1. **纯检索导向，缺乏分析结论**：现有平台多为“黄页型”词典，只能查询单一游戏的数据
2. **底层数据集缺陷**：Kaggle 等开源数据集通常样本量过小，无法真实反映长尾市场的全貌，极易导致幸存者偏差与分析过拟合；同时，官方 API 本身**不直接提供玩家自定义的 Tags 列表**，导致很难做深度的品类交叉分析。

为了从数据视角复刻“**如何发现值得做成表层卖点的题材/玩法**”这一过程，我们决定从底层爬虫做起，构建一套完全自主、可复用、且覆盖 Steam 全量活跃游戏的数据与决策平台。

---

## 2. 数据工程架构：如何搞定 ~16.5 万款游戏的数据底座？
我们设计了一套高可用的**混合式自动化数据抓取与入库流水线**，成功对 Steam 全库约 16.5 万个 AppID 进行了全量摸排，并最终筛选出约 1.9 万款具有统计价值的有效商业游戏（评论数 > 200）。

### 🛠️ 混合获取方案
*   **基础数据 (API 高速拉取)**：采用官方 `appreviews` 接口，无头静默拉取每个游戏的总评论数（Review Count）与好评率（Positive Rate）。
*   **深层数据 (Store 页面爬虫)**：由于官方 API 不返回玩家自定义 Tag，我们针对通过初筛（热度达标）的游戏，直接向 Steam 商店 HTML 页面发起请求，通过 CSS Selector 精准刮取发行日期与前 5 个核心 Tag。

### ⚡ 并发提效与高可用设计
面对海量的 HTTP 请求与 Steam 严格的反爬虫限制（IP 限流），我们设计了如下策略：
*   **漏斗式初筛**：将官方 API 作为第一道廉价漏斗，只有评论数 > 200 的有效游戏才会被放入“待深度爬取”的队列，极大地节省了网页请求的开销。
*   **SQLite 状态持久化**：引入本地 SQLite 数据库作为缓冲队列。每个游戏的爬取状态都会记录 `last_updated`，实现断点续传。无论断网还是关机，重启后脚本都会自动从上一个成功点继续，不浪费一秒钟。
*   **多分片并行执行**：为了打破单线程瓶颈，我们将全量 AppID 通过稳定的 `hash(appid) % 5` 算法切分为 5 个完全互斥的子数据库（Shard）。通过开启 5 个并发后台守护进程（Daemon Worker）同时抓取，在安全请求频率下，将原本需要一星期的爬取时间压缩到了十几个小时。

最终产生的数据集被清洗并合并为一份高价值基准库，导入至云端 **Supabase (PostgreSQL)**，供前端秒级查询。该爬虫 Workflow 代码已完全解耦并模块化，未来每半年/季度运行一次即可完成全量自动化更新。

## 3. 前端决策引擎设计
基于 Streamlit 搭建的前端，我们在 V2 版本中完成了从“底层数据看图”到“宏观业务看表”的体验重构，划分为**两大决策场景**：

### 🌟 模块一：机会发现
**回答问题：“当前有哪些值得优先研究的方向？”**
*   **当前机会**: 宏观扫描市场，发现高口碑低供给的蓝海、高热度红海、高增益组合以及近期升温最快的品类。
*   **底层数据分析**：供多维度的全局数据透视。通过综合分析、单/多 Tag 分析、破圈度 (Lift)、组合增益 (Synergy) 和时间趋势，深度挖掘 Tag 背后的市场价值和规律。

### 🎯 模块二：方向研究
**回答问题：“我想做的这个方向/组合，到底靠谱吗？”**
   **单 Tag 画像**: 深入剖析单个 Tag 的市场表现、四象限分布、协同关系及代表作。
   **多 Tag 方向验证**: 验证 2-3 个核心 Tags 组合的市场潜力，从口碑、热度、协同增益等维度进行综合评分。
   **案例库/对标**: 搜索特定游戏，查看其核心数据，并寻找共享核心 Tag 的相似游戏进行对标分析。
